# Integration Testing

## Learning Objectives

- Distinguish integration testing from unit testing in the context of an ASP.NET Core Web API.
- Understand how `WebApplicationFactory<T>` boots the full application pipeline in-process for tests.
- Write HTTP-level integration tests using the built-in `HttpClient` provided by the factory.
- Configure an in-memory or SQLite database for test isolation.
- Test authenticated endpoints by supplying JWT tokens or custom authentication handlers in the test host.

---

## Why This Matters

Unit tests verify isolated logic -- a service method, a validator, a mapping function. They are fast and deterministic. However, they cannot catch bugs that live in the wiring: middleware ordering, routing mismatches, incorrect DI registrations, or serialization behavior. Those bugs only surface when the full pipeline runs end-to-end.

Integration tests fill that gap. For a production-ready API -- which is the goal of this week's epic -- you need confidence that a real HTTP request will travel through middleware, reach the correct controller action, execute service logic, interact with the data layer, and return the expected status code and response body. A single failing integration test has higher diagnostic value than several passing unit tests when the root cause is pipeline configuration.

## The Concept

### Unit Testing vs. Integration Testing

| Concern | Unit Test | Integration Test |
|---|---|---|
| Scope | A single class or method | The full HTTP pipeline |
| Database | Mocked or in-memory stub | In-memory or SQLite test DB |
| Speed | Very fast (milliseconds) | Slower (tens of milliseconds) |
| What it catches | Logic bugs | Wiring, routing, middleware bugs |
| Typical framework | xUnit / NUnit + Moq | xUnit + `WebApplicationFactory` |

Unit tests use test doubles (mocks, fakes, stubs) to replace all external collaborators. Integration tests replace external infrastructure (a real SQL Server) with fast substitutes but keep the rest of the pipeline real.

### `WebApplicationFactory<T>`

`WebApplicationFactory<T>` lives in the `Microsoft.AspNetCore.Mvc.Testing` NuGet package. It spins up your application in-process -- meaning no network port is allocated and no actual HTTP socket is used. It creates a `TestServer` that processes `HttpRequest` objects directly in memory.

The generic parameter `T` is typically your application's entry point, which in modern .NET 6+ minimal hosting is the `Program` class itself.

```csharp
// Test project: MyApi.Tests.csproj
// Reference: <PackageReference Include="Microsoft.AspNetCore.Mvc.Testing" Version="8.*" />
// Reference your main API project.

public class ProductsControllerTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;

    public ProductsControllerTests(WebApplicationFactory<Program> factory)
    {
        _client = factory.CreateClient();
    }

    [Fact]
    public async Task GetProducts_Returns200AndList()
    {
        var response = await _client.GetAsync("/api/products");

        response.EnsureSuccessStatusCode(); // throws if not 2xx
        var body = await response.Content.ReadAsStringAsync();
        Assert.Contains("\"id\"", body);
    }
}
```

### Making `Program` Accessible

The `Program` class generated by the minimal hosting model is internal by default. Expose it to the test project by adding the following to your main project's `Program.cs` (or a separate file):

```csharp
// Program.cs -- add at the very bottom, outside any method
public partial class Program { }
```

Then ensure the test project references the API project:

```xml
<!-- MyApi.Tests.csproj -->
<ItemGroup>
  <ProjectReference Include="..\MyApi\MyApi.csproj" />
</ItemGroup>
```

### Customizing the Test Host

`WebApplicationFactory<T>` supports a `WithWebHostBuilder` override that lets you swap out services or configuration for the test environment. This is the standard way to replace the production database with a fast in-memory or SQLite database.

```csharp
private HttpClient CreateClient()
{
    var factory = new WebApplicationFactory<Program>()
        .WithWebHostBuilder(builder =>
        {
            builder.ConfigureServices(services =>
            {
                // Remove the real DbContext registration.
                var descriptor = services.SingleOrDefault(
                    d => d.ServiceType == typeof(DbContextOptions<AppDbContext>));

                if (descriptor != null)
                    services.Remove(descriptor);

                // Add an in-memory database unique per test run.
                services.AddDbContext<AppDbContext>(options =>
                    options.UseInMemoryDatabase("IntegrationTestDb"));
            });
        });

    return factory.CreateClient();
}
```

> **Note on `UseInMemoryDatabase`:** The EF Core in-memory provider is convenient but does not enforce relational constraints (foreign keys, unique indexes). For constraint testing, prefer SQLite in-memory mode (`options.UseSqlite("Data Source=:memory:")`). For most pipeline tests the in-memory provider is sufficient.

### Testing Validation and Error Paths

Integration tests should cover both success paths (happy path) and error paths. This is where they deliver the most value because the full `[ApiController]` automatic validation pipeline runs.

```csharp
[Fact]
public async Task PostProduct_WithInvalidBody_Returns400()
{
    // Name is required; sending an empty object triggers model validation.
    var payload = new StringContent(
        "{\"name\": \"\", \"price\": -5}",
        Encoding.UTF8,
        "application/json");

    var response = await _client.PostAsync("/api/products", payload);

    Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);
}
```

### Testing Authenticated Endpoints

When endpoints are protected with `[Authorize]`, your test client must supply credentials. There are two practical approaches:

**Option 1 -- Generate a real JWT in the test setup.** Use the same `JwtSecurityTokenHandler` your production code uses, generate a token with the correct issuer and audience, and add it to the `HttpClient`'s default headers.

```csharp
_client.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", GenerateTestToken());
```

**Option 2 -- Swap the authentication scheme in the test host.** Register a fake authentication handler that always succeeds and produces a known `ClaimsPrincipal`. This avoids key-management complexity in tests.

```csharp
builder.ConfigureServices(services =>
{
    services.AddAuthentication("TestScheme")
        .AddScheme<AuthenticationSchemeOptions, TestAuthHandler>("TestScheme", _ => { });
});

// TestAuthHandler.cs
public class TestAuthHandler : AuthenticationHandler<AuthenticationSchemeOptions>
{
    protected override Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        var claims = new[] { new Claim(ClaimTypes.Name, "testuser"), new Claim(ClaimTypes.Role, "Admin") };
        var identity = new ClaimsIdentity(claims, "TestScheme");
        var principal = new ClaimsPrincipal(identity);
        var ticket = new AuthenticationTicket(principal, "TestScheme");
        return Task.FromResult(AuthenticateResult.Success(ticket));
    }
}
```

### Assertion Patterns

Beyond status codes, assert on the response body by deserializing JSON:

```csharp
var content = await response.Content.ReadAsStringAsync();
var products = JsonSerializer.Deserialize<List<ProductResponseDto>>(content,
    new JsonSerializerOptions { PropertyNameCaseInsensitive = true });

Assert.NotNull(products);
Assert.True(products.Count > 0);
```

Check `Location` headers for `201 Created` responses:

```csharp
Assert.Equal(HttpStatusCode.Created, response.StatusCode);
Assert.NotNull(response.Headers.Location);
```

## Code Example: Full Test Class Skeleton

```csharp
public class ProductsIntegrationTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;

    public ProductsIntegrationTests(WebApplicationFactory<Program> factory)
    {
        _client = factory.WithWebHostBuilder(builder =>
        {
            builder.ConfigureServices(services =>
            {
                // Replace production DbContext with in-memory version.
                var descriptor = services.SingleOrDefault(
                    d => d.ServiceType == typeof(DbContextOptions<AppDbContext>));
                if (descriptor != null) services.Remove(descriptor);

                services.AddDbContext<AppDbContext>(options =>
                    options.UseInMemoryDatabase(Guid.NewGuid().ToString()));

                // Seed test data.
                var sp = services.BuildServiceProvider();
                using var scope = sp.CreateScope();
                var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
                db.Database.EnsureCreated();
                db.Products.Add(new Product { Id = 1, Name = "Test Widget", Price = 9.99m });
                db.SaveChanges();
            });
        }).CreateClient();
    }

    [Fact]
    public async Task GetAllProducts_Returns200()
    {
        var response = await _client.GetAsync("/api/products");
        response.EnsureSuccessStatusCode();
    }

    [Fact]
    public async Task GetProductById_NotFound_Returns404()
    {
        var response = await _client.GetAsync("/api/products/99999");
        Assert.Equal(HttpStatusCode.NotFound, response.StatusCode);
    }

    [Fact]
    public async Task PostProduct_NoAuth_Returns401()
    {
        var payload = new StringContent(
            "{\"name\": \"New Product\", \"price\": 19.99}",
            Encoding.UTF8,
            "application/json");

        var response = await _client.PostAsync("/api/products", payload);
        Assert.Equal(HttpStatusCode.Unauthorized, response.StatusCode);
    }

    [Fact]
    public async Task PostProduct_InvalidBody_Returns400()
    {
        // Add auth header here if needed.
        var payload = new StringContent("{}", Encoding.UTF8, "application/json");
        var response = await _client.PostAsync("/api/products", payload);
        Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);
    }
}
```

## Summary

- `WebApplicationFactory<T>` boots the real ASP.NET Core pipeline in-process, enabling true HTTP-level tests without a running server.
- Replace production databases with in-memory or SQLite equivalents using `WithWebHostBuilder` + `ConfigureServices`.
- Test both happy paths (200 OK, 201 Created) and error paths (400, 401, 404) to catch wiring and validation bugs.
- Authenticate test requests either by generating real tokens or by swapping in a fake authentication handler for the test host.
- Keep tests isolated: use a unique database name per test class (or per test) to prevent state leaking between runs.
