# ASP.NET Core Project Structure

## Learning Objectives

- Identify every file and folder generated by the default `dotnet new webapi` template.
- Explain the role of `Program.cs`, `appsettings.json`, and `launchSettings.json`.
- Understand how the `.csproj` file declares dependencies and target framework.
- Know where controllers, models, and static assets conventionally live.

---

## Why This Matters

Every ASP.NET Core project you join or create will follow a predictable folder layout. Knowing the purpose of each file immediately makes you productive in unfamiliar codebases. It also prevents common mistakes such as placing configuration in the wrong file, hard-coding secrets that belong in user secrets, or registering middleware after `app.Build()` — errors that stem from not understanding what each file owns.

---

## The Concept

### Scaffolded Project Layout

Running `dotnet new webapi -n MyApi` produces the following structure:

```
MyApi/
|-- MyApi.csproj
|-- Program.cs
|-- appsettings.json
|-- appsettings.Development.json
|-- Properties/
|   `-- launchSettings.json
|-- Controllers/
|   `-- WeatherForecastController.cs
|-- WeatherForecast.cs
```

Each file has a distinct responsibility.

---

### `MyApi.csproj`

The project file is the build manifest. It declares the target framework, NuGet package references, and build properties.

```xml
<Project Sdk="Microsoft.NET.Sdk.Web">

  <PropertyGroup>
    <TargetFramework>net8.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.AspNetCore.OpenApi" Version="8.0.0" />
    <PackageReference Include="Swashbuckle.AspNetCore" Version="6.5.0" />
  </ItemGroup>

</Project>
```

Key properties:

| Property | Purpose |
|---|---|
| `Sdk="Microsoft.NET.Sdk.Web"` | Pulls in all ASP.NET Core build targets and implicit package references. |
| `TargetFramework` | Specifies which .NET runtime to compile against. |
| `Nullable` | Enables nullable reference type annotations (strongly recommended). |
| `ImplicitUsings` | Auto-includes common `using` directives such as `System`, `System.Collections.Generic`, etc. |

---

### `Program.cs`

This is the application entry point and the single place where you configure both the dependency injection (DI) container and the middleware pipeline. In .NET 6+, the class and `Main` method are eliminated in favor of top-level statements.

```csharp
var builder = WebApplication.CreateBuilder(args);

// --- Service Registration Phase ---
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

// --- Pipeline Configuration Phase ---
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();

app.Run();
```

Nothing in the application should be configured in two places. If a service is not registered here, it cannot be injected anywhere in the app.

### `appsettings.json` and `appsettings.Development.json`

`appsettings.json` holds application-level configuration that is safe to commit to source control: connection strings (without passwords), feature flags, pagination defaults, logging levels.

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.AspNetCore": "Warning"
    }
  },
  "AllowedHosts": "*",
  "AppSettings": {
    "MaxPageSize": 50
  }
}
```

`appsettings.Development.json` contains values that **override** `appsettings.json` when the environment is `Development`. It is appropriate for verbose logging, development database URLs, and other settings that differ locally.

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Debug",
      "Microsoft.AspNetCore": "Information"
    }
  }
}
```

**Never place secrets (passwords, API keys, connection strings with credentials) in these files.** Use user secrets locally and environment variables or Azure Key Vault in production. You will cover configuration in depth in `environment-and-configuration.md`.

---

### `Properties/launchSettings.json`

This file is consumed by the .NET CLI and Visual Studio during local development only. It is not deployed and has no effect in production.

```json
{
  "profiles": {
    "http": {
      "commandName": "Project",
      "dotnetRunMessages": true,
      "launchBrowser": true,
      "launchUrl": "swagger",
      "applicationUrl": "http://localhost:5000",
      "environmentVariables": {
        "ASPNETCORE_ENVIRONMENT": "Development"
      }
    },
    "https": {
      "commandName": "Project",
      "applicationUrl": "https://localhost:7001;http://localhost:5000",
      "environmentVariables": {
        "ASPNETCORE_ENVIRONMENT": "Development"
      }
    }
  }
}
```

Key fields:

| Field | Purpose |
|---|---|
| `applicationUrl` | The URL Kestrel binds to during local development. |
| `ASPNETCORE_ENVIRONMENT` | Sets the environment name, controlling which `appsettings.{Env}.json` overlay is loaded. |
| `launchBrowser` / `launchUrl` | Opens the browser to Swagger on `dotnet run`. |

---

### `Controllers/`

By convention, controllers live in the `Controllers/` folder. Each controller is a class that inherits from `ControllerBase` (for APIs) and is decorated with `[ApiController]` and `[Route]`. You will cover controllers in detail in `controllers.md`.

---

### `Models/` (Convention — Not Scaffolded)

The default template does not create a `Models/` folder, but convention dictates placing domain entities, DTOs, and view models there. In larger projects this folder is further split into `Models/Domain/`, `Models/Requests/`, and `Models/Responses/`.

---

### `wwwroot/` (For MVC / Razor Projects)

Web API projects typically do not serve static files, so `wwwroot/` is absent from the default Web API template. If you add static file serving middleware (`UseStaticFiles()`), this folder becomes the root for CSS, JavaScript, and image assets.

---

## Summary

- The `.csproj` declares the framework target and NuGet dependencies.
- `Program.cs` owns both the DI service registration phase and the middleware pipeline configuration phase.
- `appsettings.json` holds environment-agnostic configuration; `appsettings.Development.json` holds local overrides. Neither should ever contain secrets.
- `launchSettings.json` is a local development tool only — it does not affect production.
- Controllers live in `Controllers/` by convention; models and DTOs live in `Models/`.
